What is QA Lab?

01

Running Tests

03

A Test Run is a session where you execute a set of test cases and record their outcomes. Each run is saved with full history so you can review pass rates over time.

Test Run Workflow

1
Name the run
2
Select test cases
3
Start run
4
Mark results
5
Save & review

Step-by-step: Starting a Test Run

1
Go to Test runs

Click "Test runs" in the project sidebar.

2
Give the run a name and build version

Examples: "Sprint 12 regression", "v2.1 smoke test". Add the build number so results are traceable.

3
Select test cases

Use filters (module, priority, tag) to narrow down cases. Click Select all or pick individually. Then click Start run.

4
Execute and mark status

Work through each case. Click the status dropdown to set Pass, Fail, Blocker, Skip, etc. For failures, you can log a bug directly from the run screen.

5
Save the run

When complete, click Save run. Results are stored and visible in Reports and the project dashboard.

Status Options During a Run

Pass — works as expected
Fail — defect found
Blocker — cannot proceed
Skipped — not in scope
Not Executed — pending

Tips for Test Runs

Assign a Test Plan to group runs by release or sprint milestone.
Drafts are saved automatically — if you close the browser mid-run, your progress is preserved.
Once a run is saved it cannot be edited. Review results before saving.

Bug Tracker

04

The Bug Tracker is where all defects are logged, prioritised and tracked to resolution. Every bug gets an auto-assigned ID like BUG-LO-003 based on its module.

Logging a Bug

1
Click "+ Log bug"

Open the Bug tracker from the sidebar and click the green button at the top right.

2
Fill in bug details

Give a clear title, choose the module, and describe steps to reproduce, expected vs actual results.

3
Set Severity and Priority

Severity is the impact of the bug. Priority is how urgently it needs to be fixed. These are independent — a cosmetic issue can be high priority for a release.

4
Link to a test case (optional)

If this bug was found during testing, link the relevant test case. This builds traceability.

Requirements, Reports & Team

05

Requirements & Coverage

Requirements track what the software must do. Linking test cases to requirements gives you automatic coverage visibility.

1
Add a requirement

Click "+ Add requirement" and describe the feature.

2
Link test cases

Open a requirement and attach the test cases that verify it.

3
Track coverage

QA Lab shows what % of requirements are covered and the pass rate per requirement.

Test Plans & Milestones

Use Test Plans to group test runs by release cycle or sprint. Use Milestones to set deadlines and track overall progress.

Example: Create a Test Plan named "v3.0 Release" and assign all sprint runs to it. The plan automatically shows overall pass rate across all linked runs.
Milestone progress updates automatically as linked runs complete — no manual tracking needed.

Team Roles

Workspace members have one of three roles that control what they can do:

RoleCan Do
QA Lead Full access — create/edit/delete projects, test cases, bugs, runs, members and settings
Tester Add and update test cases, log bugs, run tests — cannot delete projects or manage team
Viewer Read-only access — can view all data but cannot make any changes
Only a QA Lead can invite new members, change roles, or delete projects. Go to Settings → Team to manage your workspace.

Tips, Tags & Quick Reference

06

Using Tags

Tags are labels you can add to test cases and bugs to group them beyond modules. They're great for marking test types like smoke test, regression, API, or feature flags like v3-feature.

How to add a tag: In the test case or bug form, click the Tags field and type a label, then press Enter or , (comma) to add it. You can add multiple tags. Existing tags are suggested as you type.
To filter by tag: Use the Tag dropdown in the toolbar on the Test cases or Bug tracker page. Click any tag chip in the table to instantly filter to that tag.

Pro Tips

⚡
Bulk status updates

Select multiple test cases using the checkboxes, then use the bulk status bar that appears to set all of them to Pass or Fail in one click.

📋
Shared Steps Library

Save repeated step sequences (like "Log in as admin") as a Shared Step group. Reuse them across test cases so you only need to update them once.

📤
Bulk upload test cases

Go to Test cases → Bulk upload. Download the CSV template, fill it in with your team, then upload all cases at once — no manual entry needed.

📥
Import JUnit XML

If your team runs automated tests, go to Test runs → Import JUnit XML to bring automation results directly into QA Lab without manual entry.

🔁
Clone a test case

Click the copy icon on any test case row to create a duplicate. Useful when tests are similar with small variations.

🗄️
Backup your data

Go to Backup in the top navigation to download a full JSON export of all your workspace data. Store it somewhere safe.

Quick Reference — ID Formats

ID FormatMeaningExample

IDs are assigned automatically when you save. The module prefix is the first 2 letters of the module name. You do not need to type IDs manually.
